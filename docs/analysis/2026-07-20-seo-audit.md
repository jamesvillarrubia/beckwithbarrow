# SEO Audit — beckwithbarrow.com

**Date:** 2026-07-20
**Scope:** Technical SEO audit only (no code changes). Live site + source inspection.
**Stack:** React SPA (Vite) on Vercel, content from Strapi, images from Cloudinary.

## Method
- Raw-HTML fetch of every top route (`curl`) to inspect server-delivered `<head>`.
- `robots.txt`, `sitemap.xml`, and JSON-LD checks against the live site.
- Source inspection of routing, `<head>` management, and image `alt` handling.

---

## Executive summary
The site has a reasonable static baseline (title, description, OG tags, perf preconnects) but is missing the fundamentals that matter for a multi-page portfolio: **per-page metadata, a real sitemap, structured data, and canonicals.** Because it's a client-rendered SPA with a single static `<head>`, every URL currently presents to search engines as the *same page*. Highest-leverage fixes are P0/P1 and none require the risky prerendering change.

---

## P0 — High impact, do first

### 1. Every route serves identical `<title>` and `<meta description>`
Evidence (raw HTML, all returned 200 with the **same** head):
| Route | Title | Description |
|---|---|---|
| `/`, `/about`, `/approach`, `/press`, `/connect`, `/projects` | `Beckwith Barrow Interior Design` | `A full-service, boutique, upscale Interior Design firm…` |

There is no `react-helmet`/head manager; `<head>` is hardcoded in `frontend/index.html`. Search engines see one title/description for the entire site, so interior pages can't rank for their own intent (e.g. "interior design press / portfolio / Berkshires").
**Fix (later):** per-route `<title>`/`<meta description>` via `react-helmet-async` (or prerender). Unique, keyword-relevant copy per page.

### 2. No real `sitemap.xml`
`GET /sitemap.xml` returns **HTTP 200 but HTML** (the SPA shell) — there is no `.xml` file in `frontend/public/`, so Vercel's SPA rewrite serves `index.html`. A false 200 of HTML is worse than a 404 for crawlers.
**Fix (later):** generate a static `frontend/public/sitemap.xml` (build-time, from the same Strapi data the static-data script already fetches) listing all pages + project/press detail URLs; reference it from `robots.txt`.

### 3. No structured data (JSON-LD)
`application/ld+json` count on homepage: **0**. For a location-based design firm, `LocalBusiness`/`Organization` schema (name, areas served — Berkshires & Boston, logo, sameAs social links) is a strong local-SEO and rich-result win.
**Fix (later):** inject `LocalBusiness` + `WebSite` JSON-LD (site-wide) and `Article`/`ImageObject` on press pages.

## P1 — Meaningful, moderate effort

### 4. No canonical tags
No `<link rel="canonical">` on any route. Combined with the SPA + potential trailing-slash/query variants, this risks duplicate-content dilution.
**Fix (later):** self-referencing canonical per page (pairs with #1).

### 5. `robots.txt` is empty (comments only) and has no `Sitemap:` line
Current file is just commented-out `Disallow` examples. Functionally "allow all" (fine), but it should positively declare crawlability and point to the sitemap once #2 exists.

### 6. Per-page Open Graph / Twitter cards
`og:title`/`og:description`/`og:image` are global (one image: `demkxyyquntlquyz4wgf.jpg`). Every shared link (Press article, project) shows the same card. No `twitter:card` tags observed.
**Fix (later):** per-page OG (title/description/image = that page's hero) + `twitter:card=summary_large_image`. Pairs with #1.

---

## P2 — Polish / content

### 7. Keyword-stuffed `meta keywords`
`meta keywords` repeats "Interior Design Studio/Firm/Company/Services" ~4×. Ignored by Google; reads as spam and clutters. Remove or reduce to a short honest list.

### 8. Image `alt` quality depends on CMS fields
Plumbing is **correct** — components use `alternativeText || <fallback>` (e.g. `ProjectBlock.tsx:138`, `HomePage.tsx:174`, `PressArticlePage.tsx`). But where Strapi `alternativeText` is empty, users get generic fallbacks ("Gallery image", "About image"). For a photo-driven portfolio, descriptive alts are both SEO and accessibility value.
**Fix (content, needs CMS):** populate `alternativeText` on portfolio/press assets in Strapi.

### 9. SPA rendering (context, not a required fix)
Content isn't in the initial HTML; Google renders JS but indexing is slower/less reliable than prerendered HTML. Prerendering/SSG (e.g. `vite-react-ssg`) would deliver #1/#3/#6 in static HTML — best-case SEO, but a real change to how the site builds. Deferred by choice; P0/P1 above capture ~80% of the value without it.

---

## What's already good (keep)
- Valid static `<title>`, `<meta description>`, `<meta robots>` baseline.
- `og:title/description/image/url` present (just not per-page).
- Performance hints: `preconnect` to Cloudinary + Google Fonts, non-blocking font load, font `preload`.
- Images optimized via Cloudinary transforms (no raw originals leaked); `alt` wired through `alternativeText`.
- `robots.txt` is a real served file (not caught by the SPA rewrite).

## Follow-up: heading-order beyond the homepage (added 2026-07-21)

The homepage `heading-order` finding is **fixed** on `chore/overnight-cleanup`
(h1 added to the hero wordmark, quote h3→h2, `ProjectBlock` h5→h3) and is now
asserted by tests via `frontend/src/utils/headingOrder.ts`.

Lighthouse only audited the homepage. A source sweep found the same class of
problem on other routes, left unfixed because each needs a human judgment about
which heading is the true page title:

| Page | Issue |
|---|---|
| `PressPage.tsx:133,163` | two `<h1>` on one page |
| `PressArticlePage.tsx:155,254` | two `<h1>`; `BlockRenderer.tsx:157` can emit a third from CMS content |
| `ProjectPage.tsx:247→267` | `h1` followed by `h3` (skips `h2`) |
| `ApproachPage.tsx:324→355` | `h2` followed by `h3` is fine, but the section under `h1` starts at `h2` only after a gap — verify visually |

`ErrorMessage.tsx:13` also renders a bare `h3`, which will violate ordering
wherever it mounts as the first heading.

Recommended: apply `findHeadingOrderViolations` in a test per page, the way
`HomePage.headings.test.tsx` does, then fix what it reports.

## Suggested sequence if/when implementing
1. `react-helmet-async` → per-page title/description/canonical/OG (#1, #4, #6).
2. Build-time `sitemap.xml` + `robots.txt` `Sitemap:` line (#2, #5).
3. Site-wide `LocalBusiness`/`WebSite` JSON-LD; `Article` on press (#3).
4. Clean `meta keywords` (#7).
5. Content pass: fill Strapi `alternativeText` (#8).
6. (Optional, separate project) prerendering/SSG (#9).
